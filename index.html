Project Overview

  • Synopsis:
    • The goal of this project is to develop a tool which simplifies and formalizes the process of creating identifiers for programmers. The tool will analyze a user’s code and make suggestions as to how they can improve their naming practices to make the code more readable, thus leading to increased developer productivity and better accessibility of software development as a field.
  • Sponsoring Organization:
    • Source Code and Natural Language Analysis Laboratory (RIT SE Dept)

  • Development Methodology:
    • The team will be using Kanban for its development methodology. Kanban fits the scope and size of the project while including a quick demo and feedback cycle. Additionally, Kanban provides a high level of visibility so the project sponsor can easily see the current state of development by glancing at a board.

Metrics

  • Process Metrics:
    • 1. Tickets completed per week
      • This will give us a rough estimate of productivity for each team member. It’s not completely accurate since each ticket will take a different amount of time to complete, but we will attempt to make tickets roughly the same amount of work.
    • 2. Number of bug issues open per week
      • This will be a rough estimate of the quality of our code. We will need to make sure that we create tickets to address these bugs so they get resolved. To keep track of which tickets are bugs on our board, we will tag them appropriately.
    • 3. New bugs opened per week
      • This will be a rough estimate of the improved quality of our code. New bugs may indicate that whatever lesson learned from other bugs was not considered, nor was the right way of programming. Bugs will be tagged appropriately.
    • 4. Bugs fixed
      • This will be a rough estimate of the maintenance of our code. This will show at what rate we can identify and fix issues in the program.
  • Project Metrics:
    • 1. Recommendation accuracy on a standard dataset
      • This will tell us how well we have implemented the rules and recommendations. It will also give us an estimate of how complete the project is.
    • 2. Time until recommendation on a standard dataset
      • These will give us an estimate of our code’s performance on a standardized test dataset. We should keep track of how much of the total time is spent waiting on the part of speech tagger to run, since that will most likely be the most time consuming although easiest to swap out as needed.
